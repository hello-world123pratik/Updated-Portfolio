SplitWise — Group Expense Splitter

Split bills with friends and settle debts with fewer payments.

↗ Live App GitHub
The Problem

When a group goes on a trip, money gets messy fast. Someone pays for dinner, someone else covers the hotel, and by the end no one knows who owes what. Most apps just record expenses — they don't tell you how to settle them. This app does both, and it calculates the fewest transactions needed to clear all debts.

What It Does
Smart algorithm that reduces the number of payments needed to settle all debts
Live updates — when someone adds an expense, everyone sees it right away
Split by equal shares, exact amounts, or percentages
Secure login with tokens — no password stored as plain text
Charts showing monthly spending and spending by category
Works on phone, tablet, and desktop — dark and light mode included
Tech Used
React.js + Tailwind — screen Node.js + Express — server MongoDB — storage Socket.io — live updates Chart.js — graphs JWT + bcrypt — login security
Key Decisions
Why use tokens (JWT) instead of regular sessions?
Tokens don't need a shared memory between servers. This means the app scales easily and works across multiple deployments without any extra setup.
How does the debt algorithm work?
Instead of tracking every transaction, I calculate each person's final balance (what they paid minus what they owe). Then I match the person who owes the most with the person owed the most — this keeps the number of payments as small as possible.
Why add live updates to a money app?
In a group setting, multiple people can add expenses at the same time. Without live updates, you'd see old data and balances would be wrong until you refresh. Real-time sync keeps everyone on the same page.

JobBoard — Job Portal with Roles

A full job portal for job seekers, employers, and admins — each with their own view.

↗ Live App GitHub
The Problem

A job platform needs to serve three completely different types of users — job seekers browsing listings, employers posting roles, and admins keeping things clean. Each group needs different screens, different permissions, and different controls. Building all three in one app cleanly is harder than it sounds.

What It Does
Three user roles: job seeker, employer, admin — each with their own dashboard
Employers can post jobs, edit them, and see who applied
Jobs only go live after an admin approves them
Users can upload resumes, photos, and company logos
Pages are locked by role — wrong users can't access the wrong areas
Deployed and running live on Render
Tech Used
Django Templates — screens Django ORM — data handling SQLite — storage Django Auth — login & roles Render — live deployment
Key Decisions
How is role access enforced?
Every page checks the user's role before loading. Trying to access the employer dashboard as a job seeker returns a clear "access denied" — not a crash or an empty page.
Why require admin approval before a job goes live?
This stops spam and fake listings. It gives the platform a way to maintain quality without building a complex filter system. Simple but effective.
